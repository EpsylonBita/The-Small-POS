# Live POS QA Suggestions

Date: 2026-06-20

These are non-blocking product and implementation improvements observed during live staff-flow testing. Confirmed defects were sent to Claude for fixes separately.

## Suggestions

- Sync Health now prevents `local-table-session:*` placeholders from being sent to the admin table-session UUID route, but an old orphaned close row can still wait on `Waiting for remote table session id` when the local order has no stored `table_session_id`. Consider a dev-only reconciliation action that resolves the remote session from the remote order/client order id or safely archives the orphan when the order is already paid and the table is no longer active.
- The production renderer build emits a CSS minifier warning for `.liquid-glass-modal-shell .min-w-\\[24px\\]`. Consider auditing the escaped Tailwind/arbitrary-value selector so build output stays warning-free and the intended minimum width utility is actually applied.
- The table-grid orders tab shows compact internal order numbers such as `#00014`. Fiscal receipt numbers should be absent entirely unless the organization has an acquired government-reporting integration such as myDATA; for non-fiscal organizations, label this only as an internal order number (for example `Order #00014`) so staff and QA do not mistake it for a fiscal receipt.
- Settings auto-opens the printer setup wizard over the Settings modal when no printer profile is resolved. Consider making this an explicit setup action or one-time dismissible prompt so staff can still inspect general settings without an immediate nested modal.
- Table-check item price editing opens as a proper nested blurred sheet, but the unit-price input still shows a dot-decimal value such as `16.65` in the Greek UI. Consider applying the same locale-aware money-input treatment proposed for table payment fields.
- Z Report exposes print/export actions both in the header and inside the right closeout rail. Consider clarifying whether the header actions are global utilities and the rail actions are closeout-context actions, or reduce duplication so staff do not have to choose between visually similar controls.
- Z Report order audit filters currently expose delivery, in-store, and pickup, while live data can include room-service rows. Consider making the type filters module/data-aware so hotel room-service, takeaway, or drive-through orders can be isolated when those flows exist.
- Z Report can show localized pending/unpaid order rows while the checklist says all order payments are balanced. If pending order rows are intentionally separate from payment-integrity blockers, consider wording the checklist as "no payment-integrity blockers" so staff do not read it as "no unpaid orders exist."
- Settings terminal details currently expose raw internal identifiers such as `main_isolated`. Prefer a staff-facing branch, terminal, or location display name.
- Greek staff flows still show seeded menu/category/product data in English, for example menu categories and item names. Decide whether catalog data should support localized display fields or whether test/demo seed data should be localized per organization language.
- Table-grid details can show English seeded/admin table notes such as `Bar table seeded for testing.` in the Greek staff UI. Prefer localized demo seed notes or a table-note localization policy if notes are expected to be staff-facing across languages.
- Nested overlays such as menu item notes and pickup customer details now visually open outside the container with screen blur, but they are custom portaled overlays rather than a shared registered modal primitive. A shared nested modal helper could centralize focus trapping, escape handling, and hiding background controls from the accessibility tree.
- Staff Schedule's overnight shortcut is now guarded against invalid 24h+ durations, but the button still looks actionable for normal same-day shifts where it intentionally keeps the end date unchanged. Consider disabling it or changing helper text unless the end time is at or before the start time.
- Reservations now resolve known tables to display labels like `#TP01`, but the unresolved-table fallback in source still appears to contain mojibake. Prefer a localized "No table" fallback or a plain ASCII placeholder so an unmatched table never leaks corrupted text to staff.
- Reservations create/detail overlays open as blurred app-level surfaces, but the detail side sheet does not dismiss on Escape. Consider adding it to the same topmost-dialog Escape handling used by other POS modals.
- Reservations Services now allows creating service bookings directly from the Reservations screen when the org has appointments and service catalog enabled. Consider adding availability/conflict feedback in this modal, matching Appointments, so staff can see whether the selected staff member is actually free before submitting.
- Reservations Room Create now uses a staff-facing room selector, but existing room reservation list/timeline labels still fall back to `roomId.slice(-4)` when `roomNumber` is missing. Consider resolving room display labels from the same rooms cache used by the selector so room rows never expose internal ID fragments.
- Reservations room create and table-account/payment overlays open as proper app-level blurred modals, but Escape dismissal is inconsistent: the room create modal and table account/payment overlays required explicit close controls during live QA. Consider routing these through the same close-only topmost-dialog Escape handling now used by the new service booking modal.
- The dashboard table-grid 2D floor plan now opens on visible markers, but the viewport is still shallow inside the dashboard card. Consider a taller 2D-mode minimum height or a fit-to-visible option so staff can inspect more of the plan before scrolling.
- On the dashboard table grid at the current POS window height (1282x802), table cards can place the action row below the initial viewport, as seen with the occupied #TP04 check. Consider reducing card height or giving filtered and low-card layouts a compact action-visible presentation so staff do not have to scroll to reach the primary actions.
- The table-account payment sheet now localizes displayed balances such as `Υπόλοιπο 16,65 €`, but its editable amount/tip fields still show dot-decimal values like `16.65` and `0.00` in the Greek UI. Consider locale-aware display or input masking for money fields while preserving parser support for both comma and dot input.
- Split-payment display totals are now localized, but the by-amount editable portion fields still show raw dot-decimal numeric input such as `€ 0.00` in the Greek UI. Consider a shared money-input affordance that displays localized values while still accepting both comma and dot decimal entry.
- Existing-order split payment behaves correctly up to the safe pre-confirmation point, but the by-amount editable portion fields still use dot-decimal values such as `€ 9.25` in the Greek UI. Consider reusing the same locale-aware money input work proposed for table checks and split payments.
- Existing-order customer-info editing validates required name/phone on submit, but the save button stays visually enabled while those required fields are empty. Consider disabling it until required fields are present or showing inline required-state feedback before staff scroll to the footer.
- Table-check move/merge actions use native select dropdowns for destination tables. The labels now follow the shared `#TP01`/`#T01` convention, but the opened dropdown is a bright OS-native list that breaks the dark POS modal style. Consider replacing these with the shared themed dropdown/popover pattern.
- Orders filters now expose localized accessible names for search clear, status, type, and date controls, but the native date input still exposes the browser/OS `Show date picker` sub-button and native selects still use OS dropdown chrome. Consider a shared themed select/date-filter primitive if fully localized POS-controlled filter UI is required.
- TableActionModal, ReservationForm, RoomsView's local modal helper, TableCheckManagerModal, TableCheckManagerModal's SecondarySheet helper, InventoryPage's history/adjustment modals, UsersPage's customer-details modal, SuppliersPage's import overlay, and ServiceCatalogView's add/edit/delete modals now carry custom topmost-dialog Escape/listener logic. Consider extracting a small shared hook such as `useTopmostDialogEscape(ref, isOpen, onClose)` or routing these through a shared app-level modal shell so focus, Escape, aria, and background treatment cannot drift.
- The dedicated Tables page action modal also opens as an app-level blurred overlay, but it still shows the raw header label `Τραπέζι B01` while the dashboard grid uses `Τραπέζι #TB01`. Consider routing that header through the shared table display helper for cross-surface consistency.
- The MenuModal table/customer chip now displays `Τραπέζι #TB02`, but opening its customer-details popover shows blank editable fields because it only reflects local pickup state. Consider syncing empty local fields from the selected customer or making table pseudo-customers read-only there so a blank save cannot clear the display label.
- The current test table check `ORD-21062026-00013` shows corrupted item notes (`????? ?t?�?tano tomato`) in the cart. A read-only DB check confirmed the same corrupted value is already stored in `orders.items[0].notes`, so this needs sample-data repair and/or note-input validation rather than a pure render fix.
- The status-blocker split-payment repair modal opens correctly for unpaid orders and no longer leaks raw backend errors, but its header only says "Split Payment". Consider showing the compact visible order label such as `#00008` or `ORD #00008` in the modal header/subtitle so staff can confirm which selected order they are repairing.
- Staff Shift closeout order-history cards now localize unpaid order payment badges instead of showing raw `pending`, but the badge still uses a card icon even when no payment method exists. Consider a neutral pending/clock icon or "not paid" treatment so staff do not read it as a card-payment state.
- The Appointments staff-filter fix now correctly reuses branch staff data, but it does so through the same loader that also fetches service options for the create modal. Consider splitting staff-filter loading from create-modal service loading so visiting Appointments does not fetch services unless the modal needs them.
- The Appointments create modal now shows the selected date/time and availability slots from the floating add path, but at the current desktop viewport the slot grid initially sits partly under the fixed footer until staff scroll the modal body. Consider tightening the calendar/slot layout or auto-scrolling the availability section into view after staff/service selection.
- Service Catalog now uses the shared localized minutes label on cards. Consider a small shared duration/unit display helper for staff cards so future salon, hotel, and restaurant views do not reintroduce hardcoded unit suffixes.
- Service Catalog cards and stats localize currency display (`9,50 €`), but the edit-service price input still shows dot-decimal values such as `9.50` in the Greek UI. Consider locale-aware input masking for money fields while preserving parser support for both comma and dot input.
- Staff Schedule weekly preview supports horizontal scrolling to Saturday/Sunday, but the scrollbar is hidden and the table initially shows only part of the week at 1282px width. Consider a visible scroll cue, sticky first column, or fit-to-week option.
- Staff Schedule's week-preview modal opens as a proper blurred app-level overlay, but Escape does not dismiss it while the add-shift modal does. Consider routing both through the same topmost-dialog Escape handling so modal dismissal behavior stays consistent.
- Staff Schedule existing shift cards are informational only; after creating a test shift there was no visible edit/delete/details action on the shift card. If POS staff are expected to correct schedule mistakes, add an explicit manage-shift action.
- Staff Schedule import currently opens a native Windows file picker with English labels. If the goal is a fully localized POS staff flow, consider a localized in-app import wizard that uses the shared app-level blurred modal treatment before delegating to file selection.
- Reservations timeline uses a native bright vertical scrollbar inside the dark POS surface after scrolling through hours. Consider a scoped dark/modern scrollbar treatment or a compact jump-to-reservation affordance so the timeline feels consistent with the rest of the table grid UI.
- Reservations tab badges appear to count existing reservations, not available reservable resources. In a mixed org, `Rooms 0` and `Services 0` can be read as "not available" even though the create modal still offers room/service selectors. Consider labeling the badge as bookings/reservations or adding availability/resource counts separately.
- Suppliers in the Greek UI still show English demo/contact data such as supplier names, contact names, emails, and `Roche VD supplier district`. Decide whether supplier master data should remain organization-entered proper data only, or whether demo/test org seed data should be localized for staff QA.
- The Suppliers invoices tab can show an empty invoices list while the right detail panel still shows the currently selected/first supplier. This preserves the existing invoice-tab behavior, but a product pass could decide whether an empty invoice list should instead show a no-invoice/no-selection detail state for clearer staff scanning.
- The Suppliers import wizard opens with a blank editable draft row visible while the draft counter still reads `0 γραμμές`. Consider clarifying whether the blank row is an input placeholder or counting it as a draft row only after required fields are entered.
- Inventory in the Greek UI still shows English seed item/category data such as `Butter`, `Chicken Breast`, `Gemini X Dairy`, and `Gemini X Proteins`. Decide whether inventory master data should support localized display fields or whether test/demo seed data should be localized per organization language.
- Desktop POS can be connected to an admin API instance that is older than the local source tree, as seen with the Housekeeping `status=all` contract. Consider adding explicit POS/admin API capability or version negotiation so compatibility fallbacks are intentional and visible rather than inferred from empty successful responses.
- The Add-ons page currently shows an empty state when no purchasable integrations are enabled for the org, but it does not give staff a direct next step or explain whether the admin marketplace/module purchase is required. Consider adding a concise admin-action hint or disabled CTA.
- The Loyalty page disabled state correctly blocks member/reward actions when the program is off, but it only says to enable it from admin. Consider naming the exact admin setting/module or summarizing what POS staff will be able to do once it is enabled.
- Users in the Greek UI still show English/French/German-style demo customer names, `example.com` emails, Swiss phone numbers, and delivery notes such as `Gate code 1234`. Decide whether demo/test org customer data should be localized/anonymized for staff QA or clearly treated as organization-entered sample data.
- Users row/detail `Αποκλεισμός` actions appear to perform the status change directly. Consider a lightweight confirmation modal for ban/unban actions to reduce accidental customer blocking, matching the stronger confirmation now used for address deletion.
